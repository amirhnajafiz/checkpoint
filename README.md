A small **service-account authorization service** written in Go, and a
reference for how to structure a real API service: HTTP, auth, database,
logging, tests, CI/CD, and a deployable container image.

Users sign in with Google, create **service accounts** (each with optional
key/value labels and a token TTL), and get a signed JWT per account. External
services then call one open endpoint to validate that token.

## What it demonstrates

Each concern is small, isolated, and readable on its own:

| Concern              | Approach                                                                                 | Where |
| -------------------- | ---------------------------------------------------------------------------------------- | ----- |
| **HTTP API**         | Echo with grouped routes, middleware, request validation, one centralized JSON error handler | [internal/http/](internal/http/) |
| **Auth**             | Google OAuth login + HMAC JWTs (user & service kinds), bearer-token middleware           | [internal/auth/](internal/auth/), [middlewares.go](internal/http/middlewares.go) |
| **Database**         | PostgreSQL via `pgx`, type-safe queries generated by `sqlc`, embedded SQL migrations, a transaction helper | [internal/db/](internal/db/), [internal/models/](internal/models/) |
| **Cache**            | Redis holds each service account's current token, expiring with the JWT                  | [internal/cache/](internal/cache/) |
| **Config**           | Layered: defaults → YAML → `.env` → `MAYIGOO_*` env vars (koanf)                          | [internal/config/](internal/config/) |
| **Logging**          | Structured `zap`, JSON or text, configurable level                                       | [internal/logger/](internal/logger/) |
| **Lifecycle**        | `errgroup` + signal-driven graceful shutdown, background daemons for usage & health      | [main.go](main.go), [internal/daemons/](internal/daemons/) |
| **Tests**            | Unit (in-process miniredis) + integration (testcontainers Postgres) + black-box API E2E  | [internal/testsupport/](internal/testsupport/) |
| **CI/CD**            | GitHub Actions, two lanes (fast checks + Docker integration), path-filtered              | [.github/workflows/ci.yml](.github/workflows/ci.yml) |
| **Container**        | Multi-stage build to a distroless static image; binary embeds templates & migrations     | [build/Dockerfile](build/Dockerfile) |

## Quick start

```bash
cp example/.env.example .env        # then add your Google OAuth client id/secret
docker compose up --build           # app on :5000, plus Postgres + Redis
```

Open <http://localhost:5000>, sign in with Google, and manage service accounts.

> [!NOTE]
> Login needs a Google OAuth 2.0 client. Set `OAUTH__CLIENT_ID` and
> `OAUTH__CLIENT_SECRET` in `.env` — see [Google OAuth setup](#google-oauth-setup).

## Configuration

Every setting has a built-in default; override any of them from a YAML file,
a `.env` file, or environment variables (highest precedence). Sections split on
a double underscore, e.g. `MAYIGOO_DB__SSL_MODE` → `db.ssl_mode`.

- [example/.env.example](example/.env.example) — every key with its default
- [example/config.yaml.example](example/config.yaml.example) — the same as YAML

The app logs its effective OAuth `redirect_uri` at startup, so you can copy it
verbatim into the Google Console.

<details>
<summary><b>Google OAuth setup</b></summary>

Create an OAuth 2.0 Client ID (Web application) in the
[Google Cloud Console](https://console.cloud.google.com/apis/credentials)
(APIs & Services → Credentials → Create credentials → OAuth client ID), then:

1. Add an **Authorized redirect URI** that matches `OAUTH__REDIRECT_URL`
   **exactly**. For the demo:

   ```
   http://localhost:5000/api/users/callback
   ```

   Google compares character-for-character: `http` (not `https`) for localhost,
   no trailing slash, port `5000`, `localhost` (not `127.0.0.1`). A mismatch
   yields `Error 400: redirect_uri_mismatch`.

2. Put the client id/secret in `.env` as `OAUTH__CLIENT_ID` and
   `OAUTH__CLIENT_SECRET`.

</details>

## API

| Method   | Path                        | Auth | Description                                   |
| -------- | --------------------------- | :--: | --------------------------------------------- |
| `GET`    | `/api/health`               |  —   | Dependency-health snapshot (503 if unhealthy) |
| `GET`    | `/api/users/login`          |  —   | Start the Google OAuth flow                   |
| `GET`    | `/api/users/callback`       |  —   | OAuth callback; returns a user JWT            |
| `GET`    | `/api/services/validate`    |  —   | Validate a service token (`Bearer <token>`)   |
| `POST`   | `/api/accounts`             | user | Create a service account, mint its token      |
| `GET`    | `/api/accounts`             | user | List the caller's accounts                    |
| `GET`    | `/api/accounts/:id`         | user | Get one account                               |
| `GET`    | `/api/accounts/:id/token`   | user | Get the account's cached token                |
| `PUT`    | `/api/accounts/:id`         | user | Update an account (rotates its token)         |
| `DELETE` | `/api/accounts/:id`         | user | Delete an account (evicts its token)          |

"user" endpoints require `Authorization: Bearer <user-jwt>` (from login).

```bash
# Create an account and read back its service token (USER_TOKEN from login).
curl -s -X POST http://localhost:5000/api/accounts \
  -H "Authorization: Bearer $USER_TOKEN" \
  -H "Content-Type: application/json" \
  -d '{"name":"demo","kv":{"env":"demo"}}' | jq .
```

## Testing

Two lanes, both run in CI:

```bash
go test ./...                 # unit: no Docker (Redis code uses in-process miniredis)
scripts/integration-test.sh   # integration: real Postgres via testcontainers (needs Docker)
```

Integration tests are gated behind the `integration` build tag and stand up a
throwaway PostgreSQL container with the real migrations applied, then exercise
the store, cache, and full HTTP API end to end. Shared helpers live in
[internal/testsupport/](internal/testsupport/).

## Deploy

The [Dockerfile](build/Dockerfile) produces a self-contained static binary on a
distroless base — templates and SQL migrations are embedded, so the image needs
nothing but Postgres and Redis to point at.

```bash
docker build -f build/Dockerfile -t mayigoo .
docker run --rm -p 5000:5000 \
  -e MAYIGOO_DB__HOST=... -e MAYIGOO_REDIS__HOST=... \
  -e MAYIGOO_JWT__SECRET=... \
  -e MAYIGOO_OAUTH__CLIENT_ID=... -e MAYIGOO_OAUTH__CLIENT_SECRET=... \
  mayigoo
```

Migrations run automatically at startup.

## Development

```bash
./scripts/check.sh    # gofmt, go vet, golangci-lint, unit tests
sqlc generate         # regenerate internal/models after changing SQL
```

Requires [Go 1.26+](https://go.dev/dl/) and, for the full toolchain,
[sqlc](https://sqlc.dev/) and [golangci-lint](https://golangci-lint.run/)
(ensure `$(go env GOPATH)/bin` is on your `PATH`).

## Project layout

```
main.go              # wiring, lifecycle, graceful shutdown
internal/
  http/              # handlers, routing, middleware, validation, error handling
  auth/              # JWT signing/parsing, Google OAuth
  db/                # connection, migrations, sqlc Store, tx helper
  models/            # sqlc-generated types & queries (do not edit)
  cache/             # Redis service-token cache
  config/            # layered configuration (koanf)
  logger/            # zap setup
  daemons/           # background usage + health workers
  testsupport/       # test doubles: miniredis + testcontainers Postgres
build/Dockerfile     # multi-stage → distroless image
scripts/             # check.sh, integration-test.sh, local-deploy.sh
```
